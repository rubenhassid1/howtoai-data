## 2. Build your own app right inside ChatGPT.

And I have to say, I’m impressed. It works:

Copy and paste this on your ChatGPT:

**This part is reserved for paid members only.**
If you see this, you’re one of them. Thanks for supporting me!

Bear with me. It will be technical.

Side note: You can give it to a dev and ask them to build it for you.

#### 1. What you’re actually building.

A ChatGPT app has two parts. That’s it.

Part one: a web UI. It renders in an iframe inside ChatGPT.

Part two: an MCP server. It exposes “tools” (capabilities) to ChatGPT.

If you can’t explain why your app works better *because it lives inside ChatGPT*, stop. Tighten your scope. OpenAI warns against porting your full product UI into chat.

---

#### 2. Build it: 12 steps.

**Step 0: Pick one job.**

One job. Time-bound. Conversational. Easy to visualize.

Write a “golden prompt set”: direct prompts, indirect prompts, negative prompts. You’ll reuse this for testing and metadata tuning.

**Step 1: Design your tools first.**

Tools are the contract between your MCP server and the model.

Rules:

- One job per tool. No “kitchen sink” endpoints.- Separate read tools from write tools.- Define explicit inputSchema, predictable outputs, and IDs the model can reuse.- Use model hints where applicable (readOnlyHint, destructiveHint, openWorldHint).

**Step 2: Plan your UI components.**

Plan viewer vs editor. Inline vs fullscreen. Empty states. Mobile behavior.

Key constraint: widgets are message-scoped. Each tool response that returns a widget creates a fresh widget instance tied to that message.

**Step 3: Build the MCP server.**

Your MCP server hosts the tools and returns structured content.

You need:

- An HTTPS /mcp endpoint (streaming-friendly).- Tool handlers that validate inputs server-side (assume prompt injection will hit you), return authoritative business data from server storage (not from the widget), and return structured outputs the model can reason over.

**Step 4: Build the web UI.**

Your UI is a normal web component that ChatGPT renders in an iframe. Use any framework.

Use the bridge:

- Initial render data comes from window.openai.toolOutput.- To call tools from the UI, use window.openai.callTool(...).- To persist widget UI state, use window.openai.setWidgetState(...).

OpenAI provides a UI design system + Figma library to match ChatGPT’s look-and-feel. Optional but useful.

**Step 5: Wire tools to UI templates.**

In your tool descriptors, you advertise which HTML template should render the output via _meta[”openai/outputTemplate”].

You must also define widget CSP via _meta[”openai/widgetCSP”]. Include the exact domains you fetch from.

**Step 6: Authentication (only if needed).**

If you expose customer-specific data or write actions, authenticate.

Expected approach: OAuth 2.1 that conforms to the MCP authorization spec. ChatGPT is the client. It supports dynamic client registration + PKCE.

**Step 7: State management.**

OpenAI splits state into three buckets. Treat this as non-negotiable.

- Authoritative business data: MCP server / your backend (long-lived).- UI state: inside widget instance (message-scoped, ephemeral).- Cross-session state: your backend storage (durable across conversations).

**Step 8: Security & privacy checklist.**

Minimum bar:

- Least privilege: request minimal scopes and access.- Do not put secrets/tokens in component props.- Redact PII in logs, define retention, honor deletion.- Validate every tool input server-side.- Understand iframe limitations: strict CSP, no alert/prompt/confirm, no clipboard APIs, limited framing.

**Step 9: Deploy.**

Host the MCP server and component bundle behind a stable HTTPS endpoint.

For local dev, use ngrok (or similar) to expose your local /mcp.

Warning: cold starts can interrupt streaming HTTP.

**Step 10: Connect it to ChatGPT.**

In ChatGPT:

- Settings → Apps & Connectors → Advanced settings → enable Developer mode.- Then Settings → Connectors → Create.- Fill in: Name, Description (used for discovery), Connector URL = your public /mcp endpoint.

Test in a fresh conversation by adding the connector via the “+” menu.

**Step 11: Test like you mean it.**

OpenAI’s testing priorities: tool correctness, component UX, discovery precision.

What to do:

- Unit-test each tool handler (schemas, errors, edge cases).- Use MCP Inspector (npx @modelcontextprotocol/inspector@latest) to debug raw requests/responses.- Validate in ChatGPT developer mode using your golden prompt set.- Test mobile layouts in iOS/Android apps.

**Step 12: Tune metadata until discovery is predictable.**

ChatGPT decides when to call your connector based on metadata. Treat metadata like product copy + an evaluation loop.

Do this:

- Iterate using your golden prompt set.- Write descriptions starting with “Use this when…” and explicitly say “Do not use when…” where needed.- Use enums and examples in parameter docs.- Change one field at a time. Measure precision/recall.

#### 3. Submit it: 4 steps.

**Step A: Confirm eligibility.**

You need:

- A verified OpenAI Platform organization (individual or business verification).- Owner role in that org.- A production, publicly accessible domain for your MCP server (not local or test endpoints).- A defined widget CSP that allows exactly the domains you fetch from.- A project with global data residency (EU data residency projects cannot submit).

**Step B: Pass the review bar before you click submit.**

OpenAI review is automated scans + manual review.

High-probability rejection reasons:

- Your app is a port of a website, not a focused in-chat workflow.- Tool design is sloppy (unclear descriptions, ambiguous scope) causing wrong activations.- Missing/overbroad CSP or security issues.- Auth is required but unclear, broken, or overreaching.- The core user task cannot be completed end-to-end inside chat.

Important: once published, tool names/signatures/descriptions are locked. If you want to change tools or metadata, you resubmit. Do not submit with placeholder tool names.

**Step C: Submit to the Dashboard.**

Process:

1. In the Dashboard, add MCP server details (and OAuth metadata if using OAuth).2. Confirm compliance with OpenAI policies (checkboxes in the form).3. Fill the required submission fields.4. Click “Submit for review”.

Constraints:

- One version per app can be in review at a time.- You’ll see status in Dashboard and get email notifications.

**Step D: After approval, publish.**

Once approved, click “Publish” in the Dashboard to list it in the ChatGPT Apps Directory.

Discovery starts basic (direct link or search by name). Strong utility and satisfaction may unlock better distribution later.

---

#### 4. The acceptance checklist.

Use this as your go/no-go gate:

**UX:** One clear job. Conversational entry. Minimal UI. End-to-end completion in chat.

**Tools:** Atomic. Well-typed schemas. Strong “Use this when / Do not use” descriptions. Correct hints.

**Security:** Least privilege. Safe logging. Server-side validation. No secrets in widget. CSP locked down.

**Tech:** Stable HTTPS /mcp. Streaming works. Correct error codes. Mobile tested.

**Admin:** Org verified + Owner role + global residency project.
